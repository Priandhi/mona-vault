# Self-Healing Services — Hardening + Watchdog + Safety

Patterns for service stacks where the agent has root/sudo-ish access and can accidentally break its own substrate. Built for the Mona stack (43.163.85.51) after a 2026-06-13 cascade where a 9router update killed hermes-gateway. The agent must (a) detect that, (b) auto-recover, (c) prevent recurrence via hardened service files, (d) refuse to execute self-destructive actions without confirmation.

## The 4 Patterns

### 1. Service File Audit (systemd --user)

When reviewing or writing a `[Service]` unit, check ALL of these — every one is a latent bug that doesn't surface until the worst moment.

**Audit checklist (read service file → check each line):**

| Check | What to look for | Why |
|---|---|---|
| `After=` deps exist | `ls ~/.config/systemd/user/<dep>.service` | Missing dep = latent startup failure, no error at boot |
| `After=network-online.target` not `network.target` | `Wants=network-online.target` too | `network.target` fires before DNS/route are up; service may start before reachable |
| `StartLimitIntervalSec` + `StartLimitBurst` set | NOT `=0` (which disables throttling) | Without limit, crash loop = infinite restarts; with limit (5/60s), service enters failed state after burst |
| `Environment=PATH` includes venv + `~/.local/bin` | uv, pip-installed CLIs, node_modules/.bin | Default systemd PATH is `/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin` — uv, nvm, pyenv installs unreachable |
| `Environment=VIRTUAL_ENV=` matches ExecStart venv | `Environment=HERMES_HOME=` if scripts expect it | Mismatched env = silent breakage (scripts read empty vars) |
| `TimeoutStopSec=` | 30s minimum for graceful drain | Default 90s is fine; if missing, restart mid-shutdown corrupts state |
| `ExecStart` binary exists + executable | `python` vs `python3` consistency | Same binary on Ubuntu 24.04, but inconsistent if service file is generated by different tools |
| `WorkingDirectory=` matches where script reads/writes files | | relative paths break silently |
| `StandardOutput=journal` (not `file:`) | | journal is searchable; file paths grow unbounded |
| Resource limits (`MemoryMax=`, `CPUQuota=`) | Sensible bounds | runaway memory = kills host |

**Reference: mona-autonomous.service bug (Jun13):** `After=mona-bot.service` referenced a service that didn't exist. systemd warns at boot but proceeds — so the bug was silent. Fix: `After=network-online.target` + `Wants=network-online.target`.

**Reference: hermes-gateway.service bug (Jun13):** `StartLimitIntervalSec=0` + `Restart=always` + `RestartSec=5` = infinite crash loop. Watchdog would just keep restarting it. Fix: `StartLimitIntervalSec=60` + `StartLimitBurst=5` (allow 5 in 60s, then service enters failed state, manual intervention needed).

**Workflow:** Backup → patch → `systemctl --user daemon-reload` → verify still running (do NOT restart unnecessarily). Reference: `references/service-audit-checklist.md`.

### 2. Watchdog (cron + shell, no PM2)

Lightweight auto-heal loop for systemd --user services. Runs from user cron (no root, no PM2 daemon). Three jobs: (a) venv integrity test, (b) auto-repair if broken, (c) auto-restart any service that's down. Plus log rotation.

**Why cron + shell, not PM2/systemd timer:** User crontab is the lowest common denominator — works even when systemd --user is broken, when no session is active (with `loginctl enable-linger`), and when PM2 daemon is dead.

**Prerequisites (one-time, before installing cron):**
```bash
# Allow user services to persist across sessions
sudo loginctl enable-linger $USER
# Verify
loginctl show-user $USER | grep Linger  # must show Linger=yes
test -d /run/user/$(id -u)  # must exist
```

**Template: `~/.hermes/scripts/<name>_watchdog.sh`** — see `scripts/watchdog-template.sh` in this skill. Key points:
- Export `XDG_RUNTIME_DIR="/run/user/$(id -u)"` at top so `systemctl --user` works from cron
- Hardcode venv path + uv path + agent source dir as variables, NOT as `~/.hermes/...` globs
- Defensive checks: `[ ! -f "$VENV_PYTHON" ] && log && exit 1` BEFORE attempting repair
- Log rotation: >10MB → move to `.1` (preserves last rotation, prevents disk fill)
- TS function: `date '+%Y-%m-%d %H:%M:%S'` for human-readable, or ISO8601 for sortable
- Per-action logging with `>> "$LOG" 2>&1` to capture both streams
- Test exit codes: `set -u` to catch unset vars, but NOT `set -e` (we want to log and continue)

**Cron line:** `*/5 * * * * /home/ubuntu/.hermes/scripts/mona_watchdog.sh` (every 5 min is fine; less = too slow to recover, more = noise)

**Install pattern (idempotent, no duplicates):**
```bash
CRON_LINE="*/5 * * * * /home/ubuntu/.hermes/scripts/mona_watchdog.sh"
(crontab -l 2>/dev/null | grep -qF "mona_watchdog.sh") \
  || (crontab -l 2>/dev/null; echo "$CRON_LINE") | crontab -
```

**Verify the cron entry didn't break existing jobs:** `crontab -l` should show ALL old entries plus the new one at the end.

**PITFALL:** grep for `reboot` or `system shutdown` in a heredoc/command will trigger Hermes hardline block. Filter your output lines carefully when verifying crontab contents (e.g., `grep -E "watchdog|alpha"` not `grep "reboot"`).

### 3. Safety Wrapper (block self-destructive commands)

The agent has shell access. It can run `systemctl --user restart hermes-gateway` and sever its own Telegram connection. The safety wrapper makes that explicit + auditable.

**Pattern:** regex classify command → safe allow | dangerous prompt | unknown prompt. Always log decision.

**Critical implementation detail (lesson from Jun13):** When storing patterns in bash arrays, do NOT use `|` as the field separator — it conflicts with regex alternation `(a|b|c)`. Use a unique separator like `###`. Field extraction via parameter expansion:
```bash
pat="${entry%%###*}"      # everything before first ###
rest="${entry#*###}"      # everything after first ###
cat="${rest%%###*}"       # first field of the rest
```
Then split `cat|risk|expl` with `IFS='|'` (no `|` in those fields).

**Two-tier classification:**
- **Safe patterns (allowlist)** → exec immediately, no prompt, no log
  - `^ls`, `^cat`, `^echo`, `^grep`, `^ps`, `^which`, `^tail`, `^head`, `^stat`
  - `^systemctl --user (is-active|show|status)`, `^journalctl`
  - `^curl (--max-time|-s|-L|-I)`
- **Dangerous patterns (blocklist)** → prompt with timeout, log decision
  - `systemctl --user (restart|stop|kill) hermes-gateway` → SELF_SEVER
  - `systemctl --user (restart|stop) cf-tunnel-*` → TUNNEL
  - `pkill -f (hermes|autonomous|9router)` → SELF_SEVER
  - `npm install -g <package>` → NPM_GLOBAL (especially 9router, observed cascade Jun13)
  - `python* -m venv --clear` → PYTHON_VENV
  - `rm -rf ~/.hermes/(hermes-agent|venv-mona)` → FILESYSTEM CRITICAL
  - `(shutdown|poweroff|reboot)` → SYSTEM CRITICAL
  - `dd ... of=/dev/` → DISK CRITICAL

**`--confirm` flag** for non-interactive (cron, automation). Without it, requires interactive `y/N` with 10s timeout. With it, logs decision as `ALLOW_CONFIRM` and proceeds.

**Always log every decision** to `~/.hermes/logs/safety_check.log` with format:
```
[ISO_TIMESTAMP] [DECISION] cat=CATEGORY risk=LEVEL cmd="..." why="..."
```
Decisions: `ALLOW` (safe pattern), `ALLOW_AFTER_PROMPT`, `ALLOW_CONFIRM`, `BLOCK` (dangerous detected), `DENY` (user said no).

**Reference: full template** in `scripts/safety_check.sh` of this skill.

### 4. Startup Check (verify before operation)

Before starting a service or a long-running operation, verify the venv has the modules it needs. A single `ModuleNotFoundError` can cascade into a service crash loop.

**Two ways to call:**
1. `ExecStartPre=` in the service file → systemd runs it before `ExecStart`, fails the start if it exits non-zero
2. Manually before any risky operation: `startup_check.py --service mona-autonomous`

**Design rules (from implementation, see `scripts/startup_check.py`):**
- Resolve venv from priority: `--venv` arg → `MONA_VENV`/`VIRTUAL_ENV` env → `--service` mapping (mirrors `ExecStart` paths) → hardcoded default
- Test python binary exists + is executable + reports version (catches `python3.12` not installed but script calls it)
- Single subprocess call to import all modules + report versions (faster than N subprocesses)
- **stdout reserved for the final report, stderr for log lines** — critical for `--json` consumers. If logs go to stdout, JSON output gets corrupted
- Meaningful exit codes: 0=ok, 2=venv missing, 3=module missing, 4=version mismatch, 5=can't determine venv
- Output format: human (table with ✓/✗), `--json` (parseable report), `--quiet` (suppress all but errors)

**Service integration pattern:**
```ini
[Service]
ExecStartPre=/home/ubuntu/.hermes/scripts/startup_check.py --service %n --modules httpx,aiohttp,openai,pydantic
ExecStart=/home/ubuntu/.hermes/hermes-agent/venv/bin/python -m my_app
```

If `ExecStartPre` exits non-zero, systemd logs the error and refuses to start the service. No silent crash, no crash loop.

**Non-blocking variant (`-` prefix) for defensive layering:**

If `startup_check.py` itself has a bug, the `ExecStartPre` will exit non-zero and systemd will refuse to start the service — **Mona goes offline**. To prevent the gatekeeper from killing its own host, prefix the command with `-`:

```ini
ExecStartPre=-/usr/bin/python3 /home/ubuntu/.hermes/scripts/startup_check.py --quiet --log-file /home/ubuntu/.hermes/logs/startup_check.log --service hermes-gateway
ExecStart=/home/ubuntu/.hermes/hermes-agent/venv/bin/python -m hermes_cli.main gateway run --replace
```

The `-` tells systemd: ignore the exit code, just log it. Verify with `systemctl --user show <svc> | grep ExecStartPreEx` — should show `flags=ignore-failure`. Use **system python** (`/usr/bin/python3`) for the wrapper, not the venv python, so the check itself doesn't depend on the venv being intact.

**Venv verification (false-positive guard):**

The user pushed back: "Pastiin dia inspect venv path langsung, bukan rely on system python environment." The fix is the `in_venv` check inside the subprocess — for every imported module, verify `module.__file__.startswith(sys.prefix)`. If a module imports from system site-packages (broken venv mid-repair, polluted `sys.path`), the check counts as FAIL even though `importlib` returned a module.

Always expose verification data in the report so the caller can audit at a glance:
- `subprocess_sys_prefix` — proves the subprocess was the venv python
- `subprocess_sys_executable` — full path
- Per-module `location` — the actual file the import came from
- Per-module `in_venv` — boolean sanity check

Without this, the tool gives false positives (says OK when really the system python is shadowing). The fix: see `scripts/startup_check.py` — the in-venv check is in the embedded `code` string.

### 6. False-positive trap test (mandatory verification)

A "false positive" is when the check says OK but it's lying — for example, system python's `httpx` shadowing a broken venv's. The `in_venv` check inside the subprocess reduces the risk, but you must verify with a **trap test**: deliberately create the broken state and confirm the check correctly FAILS.

**Pattern:**
```bash
# Create a venv with NO httpx
TEST_VENV=/tmp/test-venv-no-httpx
rm -rf "$TEST_VENV"
python3 -m venv "$TEST_VENV"
# Prove system python DOES have httpx (so a buggy check would false-positive)
/usr/bin/python3 -c "import httpx; print('system httpx:', httpx.__version__)"
# Run the check against the empty venv — MUST fail with exit 3
~/.hermes/scripts/startup_check.py --venv "$TEST_VENV" --modules httpx
# Expected: "No module named 'httpx'" + exit code 3
# NOT expected: silent exit 0 with "OK"
rm -rf "$TEST_VENV"
```

**Why this matters:** A check that always says OK is worse than no check. The user (and future-you) needs to trust the green checkmark. The trap test proves the failure path is real, not theoretical.

**User signal (Jun13):** "False-positive trap test itu yang paling penting — sistem python punya httpx tapi venv kosong tetap FAIL. Itu proof yang gua mau." When the user questions whether a verification actually verifies, build the trap test FIRST. Don't just claim "it works."

**Other trap tests worth running:**
- **Watchdog trap**: stop a service, run watchdog manually, confirm it restarts. Don't just claim "watchdog will recover" — prove it on a stopped service, then start it back.
- **Safety wrapper trap**: try running a dangerous command without `--confirm`, confirm the prompt blocks. Don't claim "the wrapper works" without testing the prompt itself.
- **Service unit trap**: deliberately break a path in the service file (e.g., wrong venv path), `daemon-reload`, attempt to start, confirm it fails cleanly. Don't claim "the service starts correctly" without testing the failure path.

### 5. Prove it, don't claim it (meta-pattern)

When the user questions whether something works correctly, the wrong response is to assert "yes it does" or to argue. The right response is to **expose the state that proves it**, so the user can verify themselves at a glance. Applied examples:

- "Are you really checking the venv, not system python?" → Add `subprocess_sys_prefix` and `module.__file__` to the report. User reads output, sees the venv path, trusts the result.
- "Did the fix actually take effect?" → Add a `systemctl --user show <svc> | grep <new field>` after the fix. Show the diff.
- "Is the service really running?" → Show `is-active` AND `PID` AND `start_time` — three independent signals of liveness.

The cost of adding proof output is small (5-10 lines). The cost of asking the user to trust you without proof is erosion of trust over time — they will keep asking "are you sure?" until you either prove it or break down.

## Discipline Rules (Jun13 lessons)

These are not code patterns — they're behavioral. They emerged from specific failures in this session and matter more than any technical pattern.

### Verify before claiming facts

Claude and the user both pushed back when the agent made claims about the system without verifying. Specific failures:
- "aiohttp is missing from hermes-agent/venv" — **wrong**, it was 3.14.1
- "It's been a crash loop since Jun 8" — **wrong**, Jun 8 was SIGTERM (planned kill), Jun 13 was one event
- "The VPS is Hye-Jin" — **wrong**, it was Mona's own VPS (43.163.85.51)

**Rule:** When making a factual claim about the system, run a `terminal()` command to verify BEFORE writing the claim. Cost of verification: 1-2 seconds. Cost of being wrong: user loses trust, makes wrong decisions based on bad analysis.

### Don't fabricate bugs to look thorough

The agent flagged Claude's watchdog script with a fake "Bug 1" (heredoc variable expansion). It looked plausible. It was wrong. `<<'EOF'` writes LITERAL text that bash expands at RUNTIME — that was the intended design. The agent was trying to look thorough by finding issues.

**Rule:** Better to say "looks good, no issues found" than to invent a bug. The user's trust is built on accuracy, not on volume of issues raised. If you genuinely suspect a bug, verify it experimentally (`bash -n`, dry run, test output) before claiming.

### Compress memory aggressively; never let it overflow

Memory is finite (2,200 chars on this stack). When a long entry is needed, trim verbosity ruthlessly: drop "we", "the", filler words; abbreviate dates; use **bold** for hard rules. If `add` fails on length, immediately `replace` to compress — don't keep trying variations that all fail.

**Rule:** Before any memory write, estimate length. If it would push over 2,100 chars (95% of budget), compress first. If you're already at 2,189+ chars, the write will fail. Plan compression before writing.

### Match user energy; don't lecture after being told to stop

User signals "stop explaining, just do" via:
- "gas" / "lanjut" / "ok" (terse acknowledgment)
- Frustration emojis (😭, anjir)
- Explicit: "jangan muter-muter" / "stop ngomong gak fokus"

When you get this signal, switch to: brief acknowledgment, action, terse result. No "I think we should consider..." No "let me explain why..." Just do and report.

User signals "I want to discuss, not act" via:
- "dengerin ya jangan dikerjain" / "review dulu"
- Sharing external suggestions for review

When you get this signal: analyze, don't execute. End with offer, not action.

### When the agent might kill itself, ASK first

The agent's own substrate is the services it depends on. Actions like `systemctl --user restart hermes-gateway`, `pkill -f hermes`, killing a cloudflared tunnel, can sever the user's connection TO the agent. **Default to confirmation.** Show the command, let the user run it, OR wrap with safety_check.sh and require `--confirm`.

## Pitfalls

- **Systemd hardline block** in Hermes agents: commands containing "reboot", "shutdown", or "poweroff" trigger an unconditional block. Filter output carefully when grepping crontab or service logs — use specific safe keywords.
- **Service PATH defaults are minimal:** `systemd --user` default PATH = `/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin`. uv, nvm, pyenv installs won't be found. Always set `Environment="PATH=..."` explicitly with venv bin + `~/.local/bin` first.
- **Cron doesn't have session bus:** `systemctl --user` from cron fails with "Failed to connect to bus" unless `XDG_RUNTIME_DIR="/run/user/$(id -u)"` is exported AND user has `Linger=yes`. Both required.
- **`StartLimitIntervalSec=0` means no throttling:** not "interval 0". Combined with `Restart=always`, crash loops run forever. Always set explicit burst + interval.
- **`After=mona-bot.service` is a latent bug** if mona-bot.service doesn't exist: systemd warns but proceeds. Service runs, dependency is silently missing. Audit by `ls`-ing each referenced service.
- **Heredoc with `<<'EOF'` is correct for runtime variable expansion.** Don't flag it as a bug. `<< EOF` expands at write time; `<<'EOF'` expands at script runtime. Both are valid — they do different things.
- **Bash `local` keyword can only be used inside a function.** Using `local var=...` at top-level script (e.g., inside a top-level `for` loop) errors with `local: can only be used in a function`. The `set -u` at the top of the script catches it. Fix: drop `local` for top-level variables, or wrap the logic in a function. Hit this in the safety_check.sh `--list` path: a `for` loop with `local pat=...` was outside any function.
- **First-class user signals to embed:** "jangan muter-muter" = pivot immediately, don't repeat failing approach. "lu masih nasehati gua lagi?" = hard stop on lecturing. Match energy: terse user → terse reply. "dengerin ya jangan dikerjain" / "review dulu" = analyze, don't execute — end with offer, not action.
