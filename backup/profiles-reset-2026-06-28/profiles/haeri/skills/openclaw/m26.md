# m26 — Product & Spec Workflows (NEW in v4.1)

Ubah obrolan jadi artefak yang bisa dieksekusi: idea kabur → di-"grill" jadi konkret → PRD → issues/tasks, plus TDD dan internal-comms. **Opsional.** Hulu-nya x7 (problem shaping), hilirnya m16 (coding). Tool: `tools/prd.py`.

| Capability | Input → Output |
|---|---|
| Grill-Me | idea kabur → idea tajam (interogasi terstruktur) |
| To-PRD | percakapan → dokumen PRD |
| To-Issues | PRD → daftar issue/task ter-breakdown |
| TDD superpower | spec → test dulu → impl |
| Internal-Comms | konteks → komunikasi profesional |

---

## 1. Grill-Me — flesh out idea kabur

Operator punya idea setengah jadi. Jangan langsung iya — *grill* sampai konkret:

```
- Tanya yang TAJAM, satu per satu: "siapa user persisnya? masalah apa yang DIBAYAR buat solve?"
- Tantang asumsi: "kenapa ini, bukan [alternatif lebih simpel]?"
- Tarik ke spesifik: scope, sukses keliatan kayak apa, non-goals (yang SENGAJA gak dikerjain)
- Berhenti pas idea udah bisa di-PRD-kan. Jangan grill berlebihan (lihat x7: jangan over-shape).
```

Beda dari x7 (shaping tujuan operasional umum) — Grill-Me khusus *idea produk/fitur* → siap jadi spec. Output: idea yang punya user, masalah, scope, sukses-metrik jelas.

## 2. To-PRD — percakapan → dokumen

```python
from prd import to_prd
doc = to_prd(conversation_or_notes, title="Fitur Referral")
# → markdown PRD: Problem · Goals · Non-Goals · Users · Requirements (P0/P1/P2) · Success Metrics · Risks · Open Questions
print(doc.markdown)
```

PRD bagus = ringkas tapi lengkap: masalah dulu (bukan solusi), goals terukur, non-goals eksplisit, requirement diprioritas (P0 wajib / P1 / P2), metrik sukses, risiko. Export via m8 (DOCX/PDF) kalau perlu format.

## 3. To-Issues — PRD → task breakdown

```python
from prd import to_issues
issues = to_issues(doc)   # → list {title, body, labels, estimate, depends_on}
# push ke GitHub/Linear/Jira via m6 (API) atau output markdown checklist
```

Tiap issue: judul actionable (verb), acceptance criteria, estimasi kasar, dependency. Cukup kecil buat dikerjain dalam 1 sesi (selaras m23 task-breakdown). Urutkan by dependency → handoff ke m16 buat eksekusi.

## 4. TDD superpower

```
1. SPEC → tulis test dari requirement DULU (test = spec yang eksekutabel)
2. RED  → jalankan, harus gagal (bukti test beneran nguji)
3. GREEN→ tulis kode minimal sampai test hijau
4. REFACTOR → rapikan, test tetap hijau
```

Test dulu = desain API dari sisi pemakai + jaring regresi otomatis. Combo x5 (Eval) + m16 (impl) + m25 (CI gate). Untuk migrasi (m25) TDD = pengaman wajib.

## 5. Internal-Comms — komunikasi profesional

Konteks teknis → komunikasi yang jelas buat audiens non-teknis/stakeholder: status update, incident report, decision memo, RFC. Prinsip: BLUF (bottom line up front), jujur soal risiko, satu CTA jelas. Komplemen m20 (tone) tapi format profesional/korporat.

---

## Catatan
- Alur lengkap: idea → Grill-Me → To-PRD → To-Issues → m16 (build, TDD) → m25 (CI/deploy).
- PRD/issues = artefak, gak ada gate. Push ke tracker eksternal via m6.
- Combo: m26 + x7 (shaping) + m16 (coding) + m8 (export doc) + m6 (push issues).
